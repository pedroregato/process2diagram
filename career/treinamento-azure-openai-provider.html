Configure e teste o Azure OpenAI Service como provider real

Você implementou o código em 2026-07-12 — 13 testes, tudo mockado, zero chamada real. Isso prova que a integração está correta, mas não te dá vivência. Este treinamento fecha essa lacuna: criar o recurso na Azure de verdade, configurar no P2D, e ver uma chamada real ir e voltar.

Requer conta Azure (free tier serve) Custo: centavos por teste ~30 min

§1Criar o recurso Azure OpenAI

Isto acontece fora do P2D, direto no Azure. Se você já tem um recurso pronto, pule para §2.

  1. 1
    Acesse o Azure AI Foundry (ou Azure Portal)

    Com uma conta Azure ativa, procure por Azure OpenAI ou Azure AI Foundry no portal e crie um novo recurso — escolha uma região com o modelo que você quer disponível (gpt-4o-mini é o mais barato para testar).

  2. 2
    Implante ("deploy") um modelo

    Dentro do recurso, vá em Deployments → Create new deployment → escolha o modelo → dê um nome ao deployment. Anote esse nome exatamente — é o que o P2D vai precisar, não o nome do modelo em si.

  3. 3
    Copie Endpoint e Key

    No menu do recurso, abra Keys and Endpoint. Você precisa de duas coisas: o Endpoint (formato https://<recurso>.openai.azure.com) e uma das duas Key. Guarde os dois — vai usar no P2D em seguida.

  4. 4
    Configure um alerta de orçamento

    No Azure, vá em Cost Management → Budgets e crie um alerta baixo (ex: US$5). Isso é higiene básica antes de testar qualquer API paga — e também é, coincidentemente, o mesmo raciocínio de FinOps que o P2D já pratica internamente com o cost_model.py.

Sem conta Azure ainda?

Tudo bem — não force a criação de conta agora só para marcar uma caixa. Estude o conceito via o guia do NotebookLM (arquivo separado) e volte para este treinamento quando tiver acesso. Enquanto isso, seja honesto: fale sobre a integração como "implementada e testada com testes automatizados/mocks — validação end-to-end contra a API real pendente", não como "usei em produção".

§2Configurar no P2D

Dois lugares fazem a mesma coisa: a barra lateral (rápido, sessão atual) ou Settings → Provedor e Modelo (mais visível, mesma persistência de sessão). Em qualquer um dos dois, selecione Azure OpenAI no seletor de provider.

Campos que vão aparecer (modules/session_security.py::render_extra_fields)

••••••••••••7f2a
https://meu-recurso.openai.azure.com
ex: gpt-4o-mini-prod

O campo Deployment Name é opcional só porque o P2D assume gpt-4o-mini como padrão — mas a Azure roteia por nome de deployment, não por model id. Se você chamou seu deployment de outra coisa no passo §1.2, preencha aqui com o nome exato, senão a chamada vai falhar com "deployment not found" (veja §5).

§3Testar de ponta a ponta

O caminho mais rápido para validar não é rodar o pipeline inteiro — é usar o Benchmark On-Demand, que faz 1 chamada real isolada.

  1. 1
    Abra LLM Benchmark & Telemetria → 🧪 Benchmark On-Demand

    No multiselect de providers, marque só Azure OpenAI. Em tipos de agente, marque só um (ex: minutes, é o mais barato/curto). Repetições: 1.

  2. 2
    Execute

    Clique em ▶ Executar Benchmark. Isso chama run_benchmark_call(), que tem uma rota própria para Azure — a mesma que você testou com mocks, agora contra a API real.

    services/llm_telemetry.py::run_benchmark_call()
  3. 3
    Leia o resultado

    Sucesso: latência real e contagem de tokens aparecem na tela. Erro: a mensagem retornada é a mesma que a Azure devolveu — guarde para comparar com a tabela de §5 se algo falhar.

§6Vocabulário de entrevista

deployment-based routing

Diferente da OpenAI direta (que roteia por model id), a Azure roteia por deployment — uma camada de indireção que existe para permitir controle de custo/quota por recurso da organização. Saber explicar essa diferença é um sinal concreto de que você configurou a integração de verdade, não só leu a respeito.

enterprise LLM gateway

Azure OpenAI é frequentemente escolhido por empresas não pelo modelo em si (o mesmo GPT-4o está disponível direto na OpenAI), mas pelo enquadramento contratual/compliance — dados processados dentro do tenant Azure da empresa, SLA corporativo, integração com Azure AD/RBAC.
