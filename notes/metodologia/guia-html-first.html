01
Conceito

O que é o método HTML-first

O método HTML-first é uma prática de desenvolvimento com IA onde a especificação de uma feature é escrita como um documento HTML interativo antes de qualquer linha de código de produção ser escrita.

A ideia veio de Thariq Shihipar, engenheiro do time do Claude Code na Anthropic: ao perceber que o Markdown — formato padrão de specs e documentação — é fundamentalmente plano, ele passou a usar HTML para criar especificações visuais, com mockups inline, campos editáveis, decisões clicáveis e checklists funcionais.

"O HTML não é um relatório para ler. É um contrato entre você e o Claude Code."
— Thariq Shihipar, Anthropic · Code with Claude, 2026

O princípio central é o "compute allocator mindset": 99% dos tokens gerados pela IA devem ir para planejamento, especificação e comunicação — não para código de produção. O HTML é onde você aloca esses tokens com máximo retorno.

✗ Antes — spec em Markdown
"Criar uma página de dashboard com 3 tabs: métricas, histórico e configurações. Mostrar os dados do banco."

O que o Claude Code vê: texto ambíguo. Inventa estrutura, escolhe campos, decide sozinho — e provavelmente diferente do que você queria.
✓ Depois — spec em HTML
HTML com mockup das 3 tabs, cada métrica com sua fonte de dados exata (hub.meta.tokens), campos editáveis, checklist de entrega e decisões abertas clicáveis.

O que o Claude Code vê: contrato visual sem ambiguidade. Implementa mais rápido e erra menos.
02
Fundamentos

Por que funciona — as 5 vantagens

👁
Você lê mais um HTML do que um texto plano
É psicológico: um documento visual com tabs, cores e hierarquia prende a atenção. Uma lista Markdown de requisitos não prende. Quanto mais você lê o plano, mais erros você pega antes do código.
🎯
Elimina ambiguidade de interpretação
Um mockup inline mostra exatamente o layout esperado. Uma fonte de dados anotada (hub.meta.tokens) não deixa dúvida de onde vem o dado. O Claude Code não precisa inferir — ele executa.
⚡
Decisões são tomadas antes do código, não durante
Cada "decisão aberta" no HTML é uma escolha que o Claude Code faria sozinho se não fosse forçada explicitamente. Clicar em uma opção antes de implementar é ordens de grandeza mais barato do que refatorar depois.
📁
O plano vive no repo como documentação
Commitar o HTML junto com o código significa que em qualquer sessão futura, você ou o Claude Code podem abrir o plano e ver: o que estava planejado, o que foi decidido, o que foi entregue. Sem perda de contexto entre sessões.
🔁
O LLM já sabe gerar HTML muito bem
Não há curva de aprendizado extra para o modelo. O Claude gera HTML de qualidade com uma instrução simples. O custo de criação do plano é baixo; o retorno em clareza e velocidade de implementação é alto.
03
Estrutura

Anatomia de um bom plano

Um plano HTML eficaz tem elementos obrigatórios, recomendados e opcionais. A combinação certa depende do tamanho e complexidade da feature.

Elementos do plano HTML — o que incluir
obrigatório
Fluxo de dados — de onde vem cada dado, qual é o formato, para onde vai. Ex: "AgentBPMN → BPMNModel.xml → bpmn_generator → coordenadas absolutas". Sem isso o Claude Code inventa a arquitetura de dados.
obrigatório
Decisões abertas clicáveis — qualquer escolha não feita vira um botão. Você clica antes de implementar. Uma decisão não resolvida = uma escolha silenciosa do Claude Code.
obrigatório
Checklist de entrega do projeto — os itens específicos do seu CLAUDE.md/AGENTS.md mapeados para esta feature. O Claude Code usa como lista de tarefas ordenada.
recomendado
Mockup da interface — para qualquer coisa com UI. Não precisa ser pixel-perfect. Mostra estrutura, hierarquia e quais dados aparecem onde. HTML interativo com tabs clicáveis é o ideal.
recomendado
Pitfalls aplicáveis — subset dos pitfalls conhecidos do projeto que são relevantes para esta feature. Economiza tokens do Claude Code procurando no arquivo completo.
recomendado
Esqueleto do arquivo — para novos arquivos, mostrar a estrutura de imports e funções principais. Não o código completo — só o esqueleto com comentários.
opcional
Exemplos de input/output — para agentes LLM, um exemplo real de entrada e o JSON esperado de saída. O Claude Code usa como few-shot interno.
opcional
Contexto de negócio — por que a feature existe, qual problema resolve. Útil quando as decisões de design dependem do objetivo, não apenas da técnica.
◆
Regra da complexidade: "Complexity has to earn its keep" (Thariq). Um plano de feature pequena deve levar menos de 5 minutos para gerar. Se está crescendo demais, a feature está grande demais — quebre em sub-features.
04
Processo

O fluxo completo em 5 passos

◆
Divisão fundamental: Claude.ai chat = planejamento e especificação. Claude Code = implementação. O HTML é a ponte. Nunca misture: pedir implementação no chat não tem acesso ao filesystem; pedir planejamento no Code sem contexto gera arquitetura inventada.
V
▸ Você
1 — Defina o que quer construir com precisão
Não comece pelo código. Comece pela pergunta: qual é a próxima coisa concreta a entregar? Quanto mais específico o objetivo, melhor o plano gerado. Exemplos bons: "criar a página X que mostra Y" · "refatorar o módulo Z para suportar W" · "adicionar o campo A ao agente B".
AI
▸ Claude.ai chat
2 — Gere o plano HTML
Cole o CLAUDE.md (ou a seção relevante) e use o prompt de geração de plano. O resultado é o artefato HTML. Use o gerador de prompt da Seção 7 deste guia para montar a instrução certa.
V
▸ Você — momento crítico
3 — Revise e resolva as decisões abertas
Abra o HTML no browser. Este é o passo mais importante. Você lê mais um HTML visual do que um texto plano — é exatamente por isso que o formato funciona. Resolva cada decisão aberta clicando nos botões. Corrija o mockup se não representa o que você quer. Cada decisão não resolvida aqui vira uma escolha silenciosa do Claude Code.
AI
▸ Claude Code
4 — Passe o HTML como spec de implementação
Salve o HTML no repo. Abra o Claude Code na raiz do projeto e passe a instrução apontando para o arquivo — não descrevendo em texto. Use a fórmula: claude "Implemente conforme [arquivo-plan.html]. Leia o CLAUDE.md primeiro."
✓
▸ Você
5 — Atualize o HTML e commite junto
Marque os itens do checklist como concluídos. Atualize o status da feature. Commite o HTML junto com o código de produção. Na próxima sessão, você e o Claude Code partem de um estado conhecido.
05
Prática

Iniciando uma nova especificação

Guia passo a passo para criar sua primeira spec HTML do zero. Siga na ordem — cada etapa depende da anterior.

Checklist de criação de spec — marque conforme avança
Defina o objetivo em uma frase. Se não couber em uma frase, quebre em sub-features. Exemplo: "Criar a página ValidationHub consolidando diagnósticos BPMN, scores e telemetria."
Identifique as fontes de dados. Liste cada dado que a feature precisa e de onde vem no estado do sistema (hub, banco, sessão, API). Isso vira o fluxo de dados no HTML.
Liste as decisões não tomadas. O que você ainda não decidiu? Qual formato de output? Qual fonte de dados? Onde fica o arquivo? Cada uma vira um bloco de decisão no HTML.
Gere o plano HTML no chat. Use o gerador de prompt da Seção 7, cole o CLAUDE.md do projeto, envie. Salve o HTML retornado.
Abra no browser e revise. O mockup representa o que você quer? Os dados exibidos têm a fonte certa? O checklist está completo? Algum pitfall esquecido?
Resolva todas as decisões abertas clicando nos botões. Não deixe nenhuma aberta para o Claude Code decidir.
Salve no repo em claude_guideline/plans/nome-da-feature-plan.html (ou equivalente no seu projeto).
Copie o prompt de implementação da Seção 08 do HTML gerado (com as decisões já preenchidas) e execute no Claude Code.
◆
Onde guardar os planos: crie uma pasta claude_guideline/plans/ (ou docs/plans/, .claude/plans/) no repositório do projeto. Nomeie os arquivos como nome-da-feature-plan.html. Referencie essa pasta no CLAUDE.md para que o Claude Code saiba que ela existe.

Três tipos de spec, três tamanhos:

TipoQuando usarO que incluirTamanho esperado
Feature nova Nova page, novo agente, novo módulo Tudo: mockup, fluxo de dados, esqueleto, checklist, pitfalls, decisões 200–400 linhas HTML
Refatoração Mudar lógica existente sem nova interface Before/after do fluxo de dados, o que NÃO muda, pitfalls aplicáveis 80–150 linhas HTML
Diagnóstico/bug Bug ou comportamento inesperado Reprodução, hipóteses ordenadas, módulos suspeitos 50–100 linhas HTML
06
Armadilhas

Erros comuns e como evitá-los

✗ Erro 1 — Plano grande demais
Criar um único HTML para toda a feature de uma vez. O plano cresce, fica difícil de revisar, perde o foco.

Solução: um HTML por sub-feature. Se o plano demora mais de 5 min para gerar, a feature está grande demais.
✗ Erro 2 — Deixar decisões abertas
Passar o HTML ao Claude Code sem resolver todas as decisões. O modelo escolhe silenciosamente — quase sempre diferente do que você queria.

Solução: zero decisões abertas antes de implementar. Clique em todos os botões.
✗ Erro 3 — Pedir implementação no chat
Misturar planejamento e implementação no mesmo lugar. O chat não tem acesso ao filesystem, não lê o código existente, não vê a estrutura real do projeto.

Solução: chat = plano HTML. Code = implementação. Sempre.
✗ Erro 4 — Não commitar o HTML
Usar o HTML só como rascunho descartável. Na próxima sessão, você perde o contexto de o que foi decidido, e o Claude Code também.

Solução: sempre commitar o plano junto com o código de produção.
✗ Erro 5 — HTML sem fontes de dados
Mockup bonito mas sem indicar de onde vem cada dado. O Claude Code vai buscar os dados em qualquer lugar que encontrar — não necessariamente no lugar correto.

Solução: anotar cada campo do mockup com sua fonte exata (hub.meta.tokens, não "os tokens").
✗ Erro 6 — Plano desatualizado
Commitar o HTML com o checklist vazio após a implementação. O plano deixa de ser documentação confiável.

Solução: marcar os itens do checklist como concluídos antes do commit final. 2 minutos de trabalho.
