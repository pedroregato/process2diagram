§2Perguntas sugeridas por tema

Cole direto na caixa de chat do NotebookLM. Cada uma força ele a citar a fonte específica (ele sempre linka o trecho de origem) — use isso para verificar se a resposta é rastreável, não decorada.

Fundamentos: MLOps vs. LLMOps

"Com base nas fontes externas sobre MLOps e LLMOps, e no código de services/llm_telemetry.py que subi, classifique: o que o P2D faz hoje é MLOps clássico, LLMOps, ou os dois? Justifique ponto a ponto."

Força uma resposta ancorada no seu próprio código, não uma definição genérica de manual.

"Quais das práticas clássicas de MLOps (versionamento de modelo, feature store, model registry, retraining pipeline) simplesmente não se aplicam a um sistema que só orquestra chamadas de API a LLMs de terceiros, como o P2D? Por quê?"

Essa é a pergunta que separa quem entende a diferença de quem só decorou os dois termos.

Observabilidade e FinOps para IA

"No trecho de CLAUDE.md sobre LLM Telemetry, identifique os 4 sinais que o sistema rastreia hoje (latência, erro, custo, qualidade de schema). Para cada um, qual seria o próximo nível de maturidade que uma empresa grande adicionaria?"

Usa seu próprio sistema como base e te empurra a pensar no "próximo passo" — ótimo para perguntas de "como você evoluiria isso".

"O que é FinOps aplicado a IA generativa, segundo as fontes que carreguei? Como o core/cost_model.py do P2D (que descrevi no briefing) se relaciona com essa prática?"

Conecta terminologia de mercado a uma peça de código concreta que você pode mostrar.

Azure AI Services — panorama

"Segundo a documentação da Microsoft Learn que carreguei, quais são as diferenças arquiteturais entre chamar a API da OpenAI diretamente e chamar via Azure OpenAI Service? Cite deployment, quota, e isolamento de dados."

É literalmente o que você configurou no treinamento prático — aqui você aprende o "porquê" por trás do "como".

Governança, LGPD e ética aplicada

"Com base no texto da LGPD e nos guias da ANPD que carreguei, e no código de modules/compliance/ que descrevi (detector, audit, consent), quais princípios legais essa implementação já atende e quais ainda não?"

Pergunta deliberadamente aberta — o objetivo é você conseguir listar honestamente lacunas, não só pontos fortes.

Simulação de entrevista

"Aja como um entrevistador técnico sênior de IA. Com base em tudo que subi sobre o P2D, me faça 5 perguntas difíceis sobre as decisões de arquitetura de telemetria e providers — inclua pelo menos uma pergunta de 'e se isso escalasse 100x'."

O melhor uso do NotebookLM aqui: ele conhece seu projeto real e pode fazer perguntas específicas, não genéricas de banco de perguntas.

§3Além do chat — outros recursos do NotebookLM

O NotebookLM não é só uma caixa de pergunta-resposta. Estes recursos ajudam a fixar o conteúdo de formas diferentes.

Audio Overview

Gera um resumo em formato de podcast (dois "apresentadores" discutindo o conteúdo). Bom para revisar no trajeto, sem precisar olhar tela.

Study Guide / FAQ automático

Pede um guia de estudo ou uma lista de perguntas frequentes geradas a partir das suas fontes — útil para revisão rápida antes de uma entrevista.

Mapa mental

Gera uma visualização de como os conceitos das fontes se conectam — ajuda a ver, por exemplo, onde LLMOps e FinOps se cruzam no seu próprio projeto.

Citações rastreáveis

Toda resposta do chat vem com marcadores clicáveis apontando o trecho exato da fonte — sempre clique para confirmar antes de repetir a afirmação em voz alta numa entrevista.

§4Como validar as respostas

Regra de ouro

O NotebookLM reduz alucinação por estar restrito às suas fontes, mas não elimina — ele ainda pode interpretar mal um trecho ou generalizar demais. Trate toda resposta como rascunho, não como verdade.

Antes de repetir algo numa entrevista

Clique na citação, leia o trecho original de onde a resposta veio, e pergunte a si mesmo: "isso é o que o código realmente faz, ou é o NotebookLM extrapolando o padrão de mercado sobre o meu caso específico?" — essa é a mesma disciplina de honestidade que guiou o briefing de gap analysis.
