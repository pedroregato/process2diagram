01 — Visão Executiva

O que é o Process2Diagram?

Process2Diagram é uma plataforma web open-source que automatiza a criação de documentação de processos de negócio a partir de linguagem natural. O usuário cola ou faz upload de uma transcrição de reunião e, em segundos, obtém um diagrama BPMN 2.0 interativo, um fluxograma Mermaid, um arquivo Draw.io e uma ata de reunião estruturada — tudo pronto para distribuição.

A solução elimina horas de trabalho manual em ferramentas de modelagem, reduz erros de interpretação e garante conformidade com o padrão internacional OMG BPMN 2.0 (ISO/IEC 19510). Além do diagrama, o pipeline extrai automaticamente ata de reunião com decisões e encaminhamentos, e um levantamento de requisitos com rastreabilidade de autoria.

⚡ Processamento Inteligente

Pipeline de IA com NLP + LLM especializado extrai automaticamente atores, atividades, decisões e fluxos a partir de texto não estruturado.

📄 BPMN 2.0 Nativo

XML gerado segue rigorosamente a especificação OMG. Compatível com Camunda, Bizagi, draw.io e bpmn.io — sem retrabalho de importação.

👥 Multi-LLM

Suporta DeepSeek, Claude (Anthropic), OpenAI, Groq e Gemini. Troca de provedor em tempo real — adapta custo à necessidade.

🔒 Segurança por Design

Chaves de API isoladas em memória de sessão Streamlit. Nunca persistidas em disco, logs ou variáveis de ambiente.

04 — Componente Crítico

Gerador BPMN 2.0 (bpmn_generator.py)

Coração técnico da solução. Converte o modelo semântico extraído pelo AgentBPMN em XML BPMN 2.0 válido com coordenadas absolutas e layout automático, pronto para importação em qualquer ferramenta BPM-compatível.

📍 Layout por Colunas

Ordenação topológica → atribuição de colunas globais → cálculo de largura por coluna → posicionamento absoluto (x, y, w, h) por elemento. Altura de lane calculada dinamicamente pelo conteúdo real.

⚪ Eliminação de Cruzamentos

Algoritmo single-pass com 4 heurísticas:
H2 — lane-span ≥ 2 com elementos em range
H3 — flow forward de longo alcance (≥ 320 px)
H4 — flows cross-lane retroativos (target_x < source_x)
Flows substituídos por pares throw/catch Link Event.

📊 Waypoints Inteligentes

3 modos em _build_di:
Normal — right-centre → left-centre (2 waypoints).
Empilhado — mesma coluna, sobreposição X: bottom-centre → top-centre.
Backward — source à direita do target: caminho em U (4 waypoints) 25 px abaixo dos elementos, evitando invasão visual.

🛠️ Defesa em Duas Camadas

Camada 1 — SKILL_BPMN.md: autochecagem obrigatória antes do JSON. Lista de nomes de lane proibidos.
Camada 2 — _enforce_rules(): R0 remove events; R1 zera lane de serviceTask; R1b substitui lane genérica por nome organizacional inferido; R2 redireciona loop de correção.

Função Responsabilidade
generate_bpmn_xml(bpmn)Ponto de entrada público. Executa 4 passes: process XML, pools/lanes, layout, DI interchange.
_assign_lanes(bpmn)4-step: explicit IDs → name match → flow-context inference → fallback first lane.
_assign_columns(order, flows)Topological sort + BFS de profundidade máxima de predecessores. Fix pós-processamento para lnk_catch.
_detect_crossings(flows, shapes)H2 + H3 + H4. Retorna set de flow IDs a substituir por Link Events.
_apply_link_events(bpmn)Substitui flows cruzados por pares throw/catch, registra em lanes e lane_assignment.
_build_di(diagram, plane, shapes)Emite BPMNShape + BPMNEdge com waypoints (normal / stacked / U-backward) e bounds de label.
_enforce_rules(model) agent_bpmn.pyPost-processing determinístico: R0 remove events LLM; R1 serviceTask lane=null; R1b infere lane; R2 redireciona loop.

06 — Modelo de Dados

KnowledgeHub — Fonte Única de Verdade

Dataclass Python que vive em st.session_state["hub"]. Cada agente lê apenas o seu input e escreve apenas no seu envelope. Um contador de versão garante rastreabilidade de cada mutação. 

@dataclass
class KnowledgeHub:
    version:          int                  # incrementado em cada .bump()
    transcript_raw:   str                  # transcrição original do usuário
    transcript_clean: str                  # texto normalizado pelo NLPChunker
    nlp:              NLPEnvelope          # atores, segmentos, idioma detectado
    bpmn:             BPMNModel            # steps, edges, lanes, bpmn_xml, mermaid
    minutes:          MinutesModel         # agenda, decisões, action items, participantes
    requirements:     RequirementsModel    # IEEE 830 reqs com speaker attribution
    validation:       ValidationReport     # reservado para PC2 AgentValidator
    meta:             SessionMetadata      # tokens, timing, provedor LLM

Regra de ouro: nunca instanciar um agente diretamente em app.py — sempre via Orchestrator.run(hub). Garante ordem de execução e integridade do estado.

07 — Segurança

Modelo de Segurança Session-Only

🔒 Isolamento de Credenciais

Chaves de API armazenadas exclusivamente em st.session_state (memória server-side por sessão). Descartadas ao fechar o browser.

🚫 Zero Persistência

Nenhuma chave é escrita em disco, banco de dados, log ou variável de ambiente. session_security.py é o único ponto de acesso.

🔄 Ponto Único de Acesso

get_session_llm_client(provider) retorna o cliente ao vivo ou None. Toda a UI condiciona processamento à presença do cliente.

Análise comparativa entre a transcrição original e o JSON BPMN extraído pelo pipeline, utilizando DeepSeek (deepseek-chat) como provedor LLM na fase de testes. Cenário avaliado: Cadastro de Estrutura e Organograma de Escola — reunião de 06/03/2026 com duração de 2h35min, transcrição de 4.926 linhas.

Dimensão Nota Evidência
Fidelidade ao processo ✓ 100 % Todos os 7 passos presentes, na ordem correta. Nenhum passo inventado; nenhum omitido.
Swimlanes / Atores ✓ Exata Auditoria (S01–S04) e Gestores Validadores (S05–S06) correspondem fielmente à transcrição. S07 com lane: null conforme OMG §7.4 — sistema sem nome declarado.
Gateways ✓ Correto S05 como is_decision: true (exclusiveGateway). Saídas: sim / validar cadastro e não / devolver para ajustes. Condições alinhadas com a transcrição.
Títulos e descrições ✓ Rico Campos description com detalhes completos de cada passo — formulários, botões, campos obrigatórios e critérios de validação extraídos literalmente da reunião.
Loop de retorno ✓ Correto Edge S06 → S03 modelado corretamente. Confirmado pela fala de Pedro: "devolveu para o cadastrar", referindo-se ao retorno à etapa de unidades inferiores.
Filtragem de ruído ✓ Excelente ~2h35 de conversa lateral sobre usabilidade, SLA e prioridades corretamente descartada. Apenas o núcleo do processo extraído — sem passos espúrios.
