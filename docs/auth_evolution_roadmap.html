BASELINE Estado Atual — Autenticação Local (v4.12) CONCLUÍDO ▶

Camada de autenticação atual: credenciais hardcoded com SHA-256. Funcional para uso interno, mas não escala para múltiplas empresas.

Características

  • Credenciais em modules/auth.py → USUARIOS — dict com hash SHA-256 hardcoded
  • Login único: usuário + senha (sem domínio)
  • API keys exclusivamente em st.session_state — nunca persistidas
  • Funciona sem Supabase — nenhuma dependência externa
  • Dois usuários: admin e demo

Arquivos

modules/auth.py # login_valido(), is_authenticated(), USUARIOS ui/auth_gate.py # render_login_page(), apply_auth_gate() core/session_state.py # init_session_state()
⚠️ Limitação: adicionar um novo usuário exige editar e fazer deploy do código-fonte.
A
FASE A Schema Supabase + Módulos de Tenant EM PROGRESSO ▶

Infraestrutura de dados e módulos Python para autenticação multi-tenant. Ainda não altera a tela de login — apenas prepara o backend.

Entregas

  • SQL executado no Supabase: tabelas tenants, tenant_users, tenant_config
  • Módulo modules/tenant_auth.py — login_tenant(), tenant_auth_available()
  • Módulo modules/tenant_config.py — load_all_config(), save_config(), apply_config_to_session()
  • Inserir primeiro tenant de teste no Supabase
  • Inserir primeiro usuário admin no tenant de teste

B
FASE B Tela de Login com Campo Domínio PRÓXIMA ▶

Atualizar ui/auth_gate.py para exibir o campo Domínio e roteiar pelo tenant_auth. Fallback local preservado.

Lógica de roteamento

Supabase configurado AND tenant_auth_available()? ├── SIM → formulário: Domínio + Usuário + Senha │ └─► login_tenant(domain, user, pass) │ ✓ → carrega tenant_config → session_state │ ✗ → "Domínio, usuário ou senha incorretos" └── NÃO → formulário: Usuário + Senha (comportamento atual) └─► auth.login_valido(user, pass)

Alterações em session_state após login tenant

st.session_state["_autenticado"] = True st.session_state["_usuario_login"] = user_login st.session_state["_usuario_nome"] = display_name st.session_state["_tenant_id"] = tenant_id st.session_state["_domain"] = domain_slug st.session_state["_role"] = role # 'admin' | 'user' # API keys carregadas automaticamente st.session_state["api_key_DeepSeek"] = config.get("deepseek_key", "") st.session_state["api_key_OpenAI"] = config.get("openai_key", "") st.session_state["api_key_Gemini"] = config.get("gemini_key", "") # ... demais provedores

Arquivos alterados

ui/auth_gate.py ← novo campo Domínio + roteamento tenant vs local modules/auth.py ← logout() limpa _tenant_id, _domain, _role
ℹ️ O sidebar já lê st.session_state["api_key_{Provider}"] — nenhuma alteração no sidebar é necessária. As chaves aparecem pré-preenchidas automaticamente.

FASE D Isolamento de Dados por Tenant FUTURO ▶

Filtrar projetos, reuniões e artefatos pelo tenant_id do usuário logado. Sem esta fase, todos os tenants compartilham os mesmos dados — aceitável para POC colaborativa.

SQL de migração

ALTER TABLE projects ADD COLUMN IF NOT EXISTS tenant_id UUID REFERENCES tenants(id); -- Associar projetos existentes ao tenant admin UPDATE projects SET tenant_id = ( SELECT id FROM tenants WHERE domain_slug = 'fgv' LIMIT 1 ) WHERE tenant_id IS NULL;

Mudanças na aplicação

  • core/project_store.list_projects() — adicionar filtro eq("tenant_id", tenant_id)
  • core/project_store.save_meeting() — passar tenant_id automaticamente na criação

S1
SEGURANÇA 1 Criptografia de API Keys (AES-256) QUANDO ABRIR AO PÚBLICO ▶

Atualmente as API keys ficam em plaintext no Supabase. Para produção pública, criptografar com AES-256 usando uma chave armazenada em st.secrets.

Implementação

pip install cryptography # st.secrets["enc_key"] = base64 de 32 bytes aleatórios # Geração: python -c "import os,base64; print(base64.b64encode(os.urandom(32)).decode())" from cryptography.fernet import Fernet def _cipher(): import streamlit as st, base64 key = base64.b64decode(st.secrets["enc_key"]) return Fernet(base64.urlsafe_b64encode(key)) def encrypt(value: str) -> str: return _cipher().encrypt(value.encode()).decode() def decrypt(value: str) -> str: return _cipher().decrypt(value.encode()).decode()

Mudança isolada em modules/tenant_config.py — uma linha em save_config() e load_all_config().

ℹ️ Alternativa: Supabase Vault (plano Pro) — criptografia gerenciada nativamente, sem código extra.
S2
SEGURANÇA 2 Senhas com bcrypt + Rate Limiting QUANDO ABRIR AO PÚBLICO ▶

SHA-256 puro sem salt é vulnerável a tabelas arco-íris. Para produção pública, migrar para bcrypt (ou Argon2).

Migração SHA-256 → bcrypt

pip install bcrypt import bcrypt def hash_password(senha: str) -> str: return bcrypt.hashpw(senha.encode(), bcrypt.gensalt(rounds=12)).decode() def verify_password(senha: str, stored_hash: str) -> bool: return bcrypt.checkpw(senha.encode(), stored_hash.encode())

Rate limiting de login

  • Contar tentativas falhas em tenant_users.failed_attempts + locked_until TIMESTAMPTZ
  • Bloquear conta por 15 min após 5 tentativas consecutivas
  • Log de auditoria em tabela auth_events (tenant_id, login, event, ip, ts)
⚠️ Streamlit não expõe IP do cliente nativamente. Considerar proxy/header X-Forwarded-For se hospedado atrás de load balancer.

Notas de Compatibilidade
Backward compatibility: o módulo modules/auth.py (login hardcoded) é preservado em todas as fases. Enquanto o Supabase não estiver configurado ou tenant_auth_available() retornar False, o login local continua funcionando — útil para desenvolvimento, emergências e deploy em ambientes sem Supabase.
Nenhuma alteração no sidebar: as API keys do tenant são carregadas em st.session_state["api_key_{Provider}"] no momento do login — o mesmo namespace que o sidebar já usa. Zero mudanças no ui/sidebar.py.
Dados compartilhados (POC): sem a Fase D, todos os projetos e reuniões são visíveis para todos os tenants. Para uma POC colaborativa onde o compartilhamento é intencional, isso é uma funcionalidade, não um bug.
